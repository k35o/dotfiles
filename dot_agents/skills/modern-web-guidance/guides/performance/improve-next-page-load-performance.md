# 次ページの読み込みパフォーマンスを改善する

サイトを移動するユーザーのページ読み込みパフォーマンスを向上させる最も効果的な方法の1つは、ユーザーが次に訪問しそうなページを _訪問する前に_ 読み込みを開始することです。これはSpeculation Rules APIを使った投機的読み込みという手法で実現できます。

## 仕組み

投機的読み込みは、JSON形式のスペキュレーションルールを使って、プリフェッチまたはプリレンダーできるリンクをブラウザに伝え、ユーザーがクリックしたときのページ読み込みパフォーマンスを向上させます。

ルールは、`urls` キーによるURLのハードコードリスト（リストルール）か、ページ上のリンクを見つけるために使う href とCSSセレクタのセットを含む `where` キー（ドキュメントルール）のいずれかです。

ルールには、ページをプリフェッチまたはプリレンダーするタイミングを指定する任意の `eagerness` プロパティを含められます。`eagerness` プロパティは `immediate`、`eager`、`moderate`、`conservative` のいずれかに設定できます。`immediate` は可能な限り早く投機を開始し、他のものは短時間のホバー、長時間のホバー、ページ上でのクリック開始といったユーザーシグナルをそれぞれ待ちます。

ルールは異なる eagerness 設定を組み合わせて、積極的にプリフェッチし、その後ユーザーがさらにインタラクションするとプリレンダーすることもできます。

## いつ使うか

投機的読み込みは特に静的なページ、つまりコンテンツがあまり頻繁に変わらず、ページ生成コストが安い（特にエッジでキャッシュされている）ページで有用です。動的ページにも使えますが、古いコンテンツになる可能性に注意する必要があります。

投機的読み込みは通常同一オリジンのリンクに使用されますが、より高度なオプションではクロスオリジンの投機もある程度可能です。このガイドではより一般的な同一オリジンのユースケースに集中します。

eager 寄りの投機的読み込みは、ユーザーが次に訪問する可能性が高いページ、たとえば見出し記事や、学習コースのような段階的プロセスの次ページに適しています。

eager の度合いが低い投機的読み込みは、ユーザーが次に何をするかが明確でない場合、ページ上に多くのリンクがあり、それぞれが訪問される可能性が同じくらいの場合に適しています。ホバーやページ上でのクリック開始などより多くのシグナルを待つことで、たとえ完全にプリフェッチ／プリレンダーされていなくても、次ページに前もって着手しユーザー体験を向上させられます。

同様に、プリフェッチはプリレンダーよりも保守的な選択肢で、（クライアント／サーバー両方で）使用するリソースは少ないものの、ユーザーにとっての利点も少なくなります。初期実装ではプリフェッチが良い選択であり、開発者が明示的に要求した段階でプリレンダーに拡張するのが良いでしょう。

## 使い方

スペキュレーションルールはJSON形式で、`<script type="speculationrules">` タグ内に含められます。ルールはドキュメントの `<head>` または `<body>` に含めることもでき、JavaScriptで動的に追加することもできます。

`tag` も、グローバルレベルまたはルールごとに使用できます。設定された場合、このタグは `Sec-Speculation-Tags` ヘッダーに含まれ、どの投機が行われたかをサーバー側で識別できます。

### 定義済みURLをプリフェッチする単純なURLリストルールの例

```html
<script type="speculationrules">
  {
    "tag": "product-page-speculations",
    "prefetch": [
      {
        "urls": ["/product/1", "/product/2", "/product/3"]
      }
    ]
  }
</script>
```

### ページ上のすべての同一オリジンリンクをプリレンダーする単純なドキュメントルールの例

```html
<script type="speculationrules">
  {
    "tag": "all-links-speculations",
    "prerender": [
      {
        "where": { "href_matches": "/*" }
      }
    ]
  }
</script>
```

### インタラクティブなサイト向けに、除外を含めてリンクをプリレンダーする複雑なドキュメントルールの例

```html
<script type="speculationrules">
  {
    "tag": "speculations-with-exclusions",
    "prerender": [
      {
        "where": {
          "and": [
            { "href_matches": "/*" },
            { "not": { "href_matches": "/wp-admin" } },
            { "not": { "href_matches": "/*\\?*(^|&)add-to-cart=*" } },
            { "not": { "selector_matches": ".do-not-prerender" } },
            { "not": { "selector_matches": "[rel~=nofollow]" } }
          ]
        }
      }
    ]
  }
</script>
```

### 混合ルールセットの例

この例は、すべてのリンクを積極的にプリフェッチし、さらに `moderate` の eagerness でより多くのシグナルが得られたときに同じリンクをプリレンダーするルールセットを示しています。

```html
<script type="speculationrules">
  {
    "prefetch": [
      {
        "tag": "prefetch-speculations",
        "where": { "href_matches": "/*" },
        "eagerness": "eager"
      }
    ],
    "prerender": [
      {
        "tag": "prerender-speculations",
        "where": { "href_matches": "/*" },
        "eagerness": "moderate"
      }
    ]
  }
</script>
```

## ベストプラクティス

- **DO** ユーザーが次に訪問する可能性の高いページをプリフェッチまたはプリレンダーするためにスペキュレーションルールを使用してください。
- **DO** コンテンツがあまり頻繁に変わらず、ページ生成コストが安い（特にエッジでキャッシュされている）静的サイトでスペキュレーションルールを使用してください。
- **DO** コンテンツが頻繁に変わる可能性があり、古くなる可能性が高く、ページ生成コストが高い動的ページにスペキュレーションルールを使用するときは、より注意してください。
- **DO** リストルールよりもドキュメントルールを優先してください。柔軟性があり、同じルールを複数のページで共有でき、事前にわからないページをプリフェッチ／プリレンダーするためにも使用できます。
- **DO** プリフェッチとプリレンダーのトレードオフを考慮し、ユースケースに適切なものを選択してください。プリレンダーはプリフェッチより高コストで、動的状態を表示する複雑なアプリケーションでは意図しない副作用を引き起こす可能性がより高いものの、より良いユーザー体験を提供します。判断に迷う場合は開発者にどちらが良いか確認してください。
- **DO** 異なる `eagerness` レベル間のトレードオフを考慮し、ユースケースに適切なものを選択してください。より eager な投機はより良いユーザー体験を提供しますが、より多くのリソースを使い、動的状態を表示する複雑なアプリケーションでは意図しない副作用を引き起こす可能性がより高くなります。判断に迷う場合は開発者にどちらが良いか確認してください。
- **DO NOT** ページ上のすべてのリンクを投機するなど、スペキュレーションルールを乱用しないでください。ブラウザには制限があります（eager でない投機は2件まで）。`immediate` はごく少数のリンクにのみ使用すべきです。
- **DO NOT** `/logout` や `/add-to-cart` のように状態変化を引き起こしそうなURLを投機しないでください。ドキュメントルールに含まれそうな場合はそれらを明示的に除外してください。
- **DO NOT** シングルページアプリケーション（SPA）でスペキュレーションルールを使用しないでください。スペキュレーションルールはマルチページアプリケーション（MPA）向けに設計されており、ナビゲーションごとに新しいドキュメントへ移動するブラウザのために用意されています。SPAではナビゲーションごとに新しいドキュメントへ移動しないため、スペキュレーションルールは期待通りに動作しません。

## ブラウザサポートとフォールバック戦略

投機的読み込みは新しい機能であり、そのためすべてのモダンブラウザでサポートされているわけではありません（Baseline limited availability）。

ただし、投機的読み込みはプログレッシブエンハンスメントです。エンハンスメントとして使うのは完全に安全であり、潜在的なパフォーマンス上のメリットを考えると強く推奨されます。ブラウザがスペキュレーションルールをサポートしていない場合、単純に無視されます。
